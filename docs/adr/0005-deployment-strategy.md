# ADR 0005 — Deployment Strategy: Multi-Cloud via .NET Aspire

**Date:** 2026-06-14
**Status:** Accepted

## Context

The Task Manager is a learning lab, not a production SaaS. The deployment strategy must:

1. Work out-of-the-box for local development (no cloud account required)
2. Be portable across the three major cloud providers (Azure, AWS, GCP) for students in different corporate environments
3. Not lock students into a single cloud vendor's proprietary services
4. Integrate with the CI/CD pipeline already built in Module 9 (GitHub Actions)

## Decision

We use **.NET Aspire** as the local orchestration and cloud manifest layer, with per-cloud deployment scripts for Azure Container Apps, AWS ECS Fargate, and GCP Cloud Run.

### Local development

`dotnet run --project aspire/TaskManager.AppHost` starts all services (API, frontend, PostgreSQL) and the Aspire Developer Dashboard (traces, logs, health) in a single command.

### Cloud deployment

The Aspire manifest (`aspire-manifest.json`) describes all services, their images, environment variables, and secret references in a cloud-neutral format. Per-cloud scripts (`aws/deploy.sh`, `infra/gcp/deploy.sh`) translate the manifest into platform-specific resources.

Secrets (DATABASE_URL, SECRET_KEY, CORS_ORIGINS) are injected at runtime from:
- **Azure:** Azure Key Vault via Managed Identity (no credentials in env)
- **AWS:** AWS Secrets Manager via ECS Task Role (IAM, no static keys)
- **GCP:** GCP Secret Manager via Workload Identity (no service account keys)

### Docker Compose (legacy)

Docker Compose is retained for CI (`docker-build` job) and as a fallback for students who cannot install .NET SDK. New features should be developed against the Aspire configuration.

## Rationale

### Why Aspire over plain Docker Compose for cloud?

Docker Compose does not translate to cloud deployment — it requires a separate layer (Kompose, ECS Compose, or manual conversion) for each cloud. Aspire generates a cloud-neutral manifest from the same codebase used for local development.

### Why not Fly.io exclusively?

Fly.io was the original deployment target (Module 13) but does not map cleanly to enterprise cloud environments where students will actually work (Azure Government, AWS GovCloud, GCP APAC regions). The multi-cloud approach teaches portable skills.

### Why not Kubernetes?

Kubernetes adds significant operational complexity for a three-service application. The lab teaches DevOps practices, not Kubernetes administration. Container-native PaaS services (Container Apps, ECS Fargate, Cloud Run) provide the same deployment model with managed control planes.

## CI/CD Integration

### Promotion pipeline (Module 16)

Every merge to `main` follows the same-artifact principle: one Docker image is built, pushed to GHCR tagged `sha-<commit>`, and then **promoted** through environments — never rebuilt between environments.

```
Merge to main → Build & publish once (GHCR :sha-abc1234)
    → deploy-staging  (automatic, no approval, GitHub Environment: staging)
        alembic upgrade head · health check · k6 smoke test
    → deploy-production  (requires named reviewer, GitHub Environment: production)
        same :sha-abc1234 · alembic upgrade head · health check · k6 smoke test
```

GitHub Environments enforce:
- **Protection rules** — named reviewer must approve before `deploy-production` runs
- **Environment secrets** — staging and production each have their own `SECRET_KEY` and `DATABASE_URL`; values never cross boundaries
- **Branch rules** — only `main` can deploy to either environment

### Supply chain and image governance (Modules 13–16)

The `publish.yml` pipeline includes the following steps before any deploy job runs:

1. **Build** — multi-stage Docker images for API and frontend
2. **Push to GHCR** — images tagged with `sha-<commit>` and `latest`
3. **Trivy image scan** — SARIF results uploaded to GitHub Security; exits non-zero on CRITICAL or HIGH CVEs
4. **SBOM generation** — CycloneDX JSON generated by `anchore/sbom-action` and uploaded as a build artifact
5. **Supply-chain verification (`cosign verify`)** — confirms the image was built by the GitHub Actions workflow and has not been tampered with; uses Sigstore keyless attestation with `--certificate-identity-regexp` scoped to the repository and `--certificate-oidc-issuer` set to `https://token.actions.githubusercontent.com`
6. **Cloud deploy** (gated by `if: false`) — to activate a target:
   - Remove `if: false` from the target job
   - Configure the matching GitHub Environment with a named reviewer (human approval gate)
   - Establish the cloud-specific OIDC trust (no long-lived credentials in GitHub Secrets)
7. **Post-deploy health check** — polls `/health` until 200 or times out and rolls back
8. **ZAP baseline DAST scan** (gated by `if: false`, runs after staging deploy) — passive OWASP ZAP scan against the live staging URL; creates a GitHub Issue on findings; enable alongside the staging deploy job

### IaC security gate (Module 17)

The `ci.yml` pipeline includes a `terraform-plan` job that runs on every pull request:

- **`tfsec`** static analysis (via `aquasec/tfsec-action@v1.0.0`) runs first — it requires no cloud credentials and scans all Terraform HCL for security misconfigurations. `soft_fail: true` means findings are reported but do not block the PR (set to `false` to hard-gate on HIGH+ findings).
- **`terraform plan`** (read-only, `continue-on-error: true`) authenticates to GCP via Workload Identity and posts the plan diff as a PR comment. It is informational on PRs; the actual `terraform apply` runs only in the CD pipeline on merge.

### Progressive delivery (Module 13 extension)

The same-artifact promotion model supports incremental rollout strategies on Fly.io:

- **Canary release** — deploy to a subset of instances first (`strategy = "canary"` in `fly.toml`); monitor error rate in Grafana before full rollout
- **Blue-green** — run two identical environments in parallel; switch traffic with zero downtime (`flyctl deploy --strategy bluegreen`)

These strategies are additive extensions on top of the staging→production promotion pipeline; they do not require separate cloud infrastructure.

### Supply chain hardening (Module 14)

All GitHub Actions steps are pinned to commit SHAs rather than floating tags (e.g., `actions/checkout@<sha>  # v4.1.0`). This prevents a compromised upstream tag from injecting malicious code into the pipeline. Dependabot (`.github/dependabot.yml`) opens weekly PRs to update SHA pins when new versions are released.

## Consequences

- Students must choose one cloud target for the Module 13 deployment exercise; the others remain gated by `if: false`
- The Aspire AppHost must be kept in sync with `docker-compose.yml` service definitions — both describe the same three services
- All secrets must be referenced by name (not value) in `appsettings.Development.json` — actual values are in `dotnet user-secrets` or cloud secret stores
- The `CORS_ORIGINS` environment variable must be set to the deployed frontend URL in production — it cannot be left at the default `http://localhost:5173`
- GitHub Actions SHA pinning must be maintained via Dependabot — floating `@vN` tags are a supply chain risk and will fail the `/check-secrets` skill's CI hygiene check
- Trivy hard gate means a new HIGH or CRITICAL CVE in a base image will block deployment until the image is updated — operators must have a documented patch runbook
- Staging and production environments must each have their own `SECRET_KEY` and `DATABASE_URL` — secrets must never be shared between environments (Module 16)
- The same Docker image (`sha-<commit>`) is deployed to staging and then promoted to production — the image is never rebuilt between environments; any rebuild invalidates the staging verification (Module 16)
